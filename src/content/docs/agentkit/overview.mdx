AgentKit lets your AI agents act on behalf of users across third-party applications to take actions such as sending emails, reading calendars, creating tickets, querying databases, and more. Scalekit handles the OAuth flows, token storage, and API authentication so your agent code focuses on what to do, not how to authenticate.

## Core concepts

Three building blocks make up Agent Auth:

**Connectors** are the third-party applications your agent can use — Gmail, Slack, Salesforce, GitHub, and many others. Each connector defines the authentication requirements and the tools available to your agent.

**Connections** are configurations you create once in the Scalekit Dashboard. A connection holds the credentials Scalekit needs to talk to a connector on your agent's behalf — OAuth client ID and secret, scopes, redirect URIs, and similar. One connection can serve all your users.

**Connected accounts** are per-user instances of a connection. When a user completes the authentication flow, Scalekit creates a connected account that holds their tokens and tracks their authorization state. Your agent uses a connected account to act on that user's behalf.

## How they fit together

```d2
direction: right

Dev: "You (developer)"
Users: "Your users"

Scalekit: {
  Conn: "Connection\n(credentials + scopes)"
  CA: "Connected accounts\n(per-user tokens)"
}

Provider: "Third-party app\n(Gmail, Slack, …)"

Dev -> Scalekit.Conn: "Configure once"
Users -> Scalekit.CA: "Authenticate"
Scalekit.CA -> Provider: "Execute tools"
```

The relationship flows in one direction: you configure a connection, your users authenticate against it to create connected accounts, and your agent uses those connected accounts to call tools.

## What Scalekit manages for you

- OAuth flows and token exchange for every connector
- Secure token storage and automatic refresh
- Per-user authorization state tracking
- Authenticated API proxying — your agent never handles raw tokens
