Scalekit uses a managed encryption key by default. With BYOK, you register a key from Google Cloud KMS that only your team can control or revoke. Scalekit never handles the raw key material; it calls the KMS API to encrypt and decrypt on each operation.

Set up the key in GCP (steps 1–3), then register and activate it in Scalekit (steps 4–6).

<Aside type="caution" title="You own the key lifecycle">
  BYOK gives you direct control over your encryption key. Scalekit cannot rotate or manage the GCP key on your behalf. If the key is disabled, destroyed, or IAM access is revoked, Scalekit cannot encrypt new data or decrypt existing records until you restore access.
</Aside>

## Prerequisites

- A **Google Cloud project** with the [Cloud KMS API enabled](https://cloud.google.com/kms/docs/create-encryption-keys)
- **IAM permissions** to create key rings, keys, and set key-level IAM policies
- The **Scalekit service account email**, shown in the dashboard when you select BYOK as the key type (see [step 4](#step-4-register-the-byok-key-in-scalekit))
- `gcloud` CLI installed and authenticated (for the CLI approach), or access to the [GCP Console](https://console.cloud.google.com/security/kms)

## Set up BYOK

### Step 1: Create a key ring

A [key ring](https://cloud.google.com/kms/docs/resource-hierarchy#key_rings) is a logical container for encryption keys in Cloud KMS. It sets the geographic location of your keys and groups them for IAM management. Key rings cannot be deleted or renamed once created. Choose the name and location carefully.

<Tabs syncKey="gcp-method">
<TabItem label="gcloud CLI">

```bash frame="terminal"
gcloud kms keyrings create "scalekit-kms-keyring" \
  --location "global" \
  --project "YOUR-GCP-PROJECT"
```

Replace `YOUR-GCP-PROJECT` with your Google Cloud project ID.

</TabItem>
<TabItem label="GCP Console">

1. Open [**Security > Key Management**](https://console.cloud.google.com/security/kms) in the GCP Console.
2. Click **Create key ring**.
3. Enter a **Key ring name** (for example, `scalekit-kms-keyring`).
4. Set **Location type** to **Global** (or choose a region that matches your data-residency requirements).
5. Click **Create**.

</TabItem>
</Tabs>

<Aside type="tip" title="Choose the right location">
  Use `global` if your Scalekit environment has no specific regional data-residency requirement. For region-specific compliance, choose a regional location such as `us-east1` or `europe-west1`. See [Cloud KMS locations](https://cloud.google.com/kms/docs/locations) for the full list.
</Aside>

### Step 2: Create an encryption key

Create a symmetric AES-256-GCM key inside the key ring.

<Tabs syncKey="gcp-method">
<TabItem label="gcloud CLI">

```bash frame="terminal"
gcloud kms keys create "scalekit-kms-key" \
  --location "global" \
  --keyring "scalekit-kms-keyring" \
  --purpose "encryption" \
  --project "YOUR-GCP-PROJECT"
```

The `--purpose "encryption"` flag creates an AES-256-GCM symmetric key.

</TabItem>
<TabItem label="GCP Console">

1. On the **Key Management** page, click the key ring you created.
2. Click **Create key**.
3. Enter a **Key name** (for example, `scalekit-kms-key`).
4. Set **Purpose** to **Symmetric encrypt/decrypt**.
5. Leave **Algorithm** as the default (**Google symmetric key**, AES-256-GCM).
6. Set **Rotation period** to **Never**. Rotation is managed through the Scalekit dashboard, not GCP auto-rotation.
7. Click **Create**.

</TabItem>
</Tabs>

### Step 3: Grant Scalekit access to the key

Scope both roles to the key itself, not the project. This limits Scalekit's IAM access to this key only.

| Role | Purpose |
|------|---------|
| `roles/cloudkms.cryptoKeyEncrypterDecrypter` | Encrypt and decrypt the DEK |
| `roles/cloudkms.viewer` | Read key metadata for health checks and listing |

Replace `SCALEKIT-SERVICE-ACCOUNT` with the service account email shown in the Scalekit dashboard.

<Tabs syncKey="gcp-method">
<TabItem label="gcloud CLI">

```bash frame="terminal"
gcloud kms keys add-iam-policy-binding "scalekit-kms-key" \
  --keyring "scalekit-kms-keyring" \
  --location "global" \
  --project "YOUR-GCP-PROJECT" \
  --member "serviceAccount:SCALEKIT-SERVICE-ACCOUNT" \
  --role "roles/cloudkms.cryptoKeyEncrypterDecrypter"

gcloud kms keys add-iam-policy-binding "scalekit-kms-key" \
  --keyring "scalekit-kms-keyring" \
  --location "global" \
  --project "YOUR-GCP-PROJECT" \
  --member "serviceAccount:SCALEKIT-SERVICE-ACCOUNT" \
  --role "roles/cloudkms.viewer"
```

</TabItem>
<TabItem label="GCP Console">

1. In the GCP Console, open [**Security > Key management**](https://console.cloud.google.com/security/kms), click **Key rings**, then click the key ring name.
2. Click the key name, then click the **Permissions** tab.
3. Click **Grant access**. A side panel opens.
4. In the **New principals** field, enter the Scalekit service account email.
5. In the **Assign Roles** section:
   - From the first **Role** dropdown, select **Cloud KMS CryptoKey Encrypter/Decrypter**.
   - Click **+ Add another role** and select **Cloud KMS Viewer** from the second dropdown.
6. Click **Save**. The policy update takes effect within a few minutes.

<Aside type="note" title="Key-level vs project-level IAM">
  Grant these roles at the **key level** (not at the project level) to limit Scalekit's access to only this specific key.
</Aside>

</TabItem>
</Tabs>

### Step 4: Register the BYOK key in Scalekit

Add the key to your Scalekit environment:

1. In the Scalekit dashboard, go to **Settings** and open the **Encryption keys** tab.
2. Click **Create New Key** and select **BYOK - GCP Cloud KMS**.
3. Copy the Scalekit service account email shown in the modal. You need it for [step 3](#step-3-grant-scalekit-access-to-the-key) if you have not granted IAM access yet.
4. Enter the fully-qualified GCP key resource name:

   ```
   projects/YOUR-GCP-PROJECT/locations/global/keyRings/scalekit-kms-keyring/cryptoKeys/scalekit-kms-key
   ```

   To retrieve the exact name from the CLI:

   ```bash frame="terminal"
   gcloud kms keys describe "scalekit-kms-key" \
     --keyring "scalekit-kms-keyring" \
     --location "global" \
     --project "YOUR-GCP-PROJECT" \
     --format="value(name)"
   ```

5. Click **Create**. The key is created in **Staged** state.

### Step 5: Activate the key

A staged key encrypts nothing. Click **Activate** on the staged key. The key becomes Primary and Scalekit uses it to encrypt all data from that point forward.

<Aside type="caution" title="Activation is permanent">
  Once activated, you cannot deactivate the key without creating and activating a new key. Confirm your IAM grants are in place before activating.
</Aside>

### Step 6: Re-encrypt existing data

Activation protects new writes only. To migrate existing records, click **Re-encrypt Data**. Scalekit decrypts each existing record using the previous key and re-encrypts it with the new key. The dashboard shows progress and any rows that could not be recovered.

## Monitor with audit logs

Cloud KMS logs every cryptographic operation to [Cloud Audit Logs](https://cloud.google.com/kms/docs/audit-logging). Use this filter in **Cloud Logging** to see all encrypt, decrypt, and key update events for your key:

```
resource.type="cloudkms_cryptokey"
```

## Key resource name format

The key reference you enter in Scalekit must follow this format:

```
projects/{PROJECT_ID}/locations/{LOCATION}/keyRings/{KEYRING_NAME}/cryptoKeys/{KEY_NAME}
```

Retrieve the exact key ring path with:

```bash frame="terminal"
gcloud kms keyrings describe "scalekit-kms-keyring" \
  --location "global" \
  --project "YOUR-GCP-PROJECT" \
  --format="value(name)"
```

Append `/cryptoKeys/scalekit-kms-key` to get the full key reference.

## Fix common errors

<details>
<summary>Permission denied when activating or using the key</summary>

Check that both IAM bindings are applied to the correct key:

```bash frame="terminal"
gcloud kms keys get-iam-policy "scalekit-kms-key" \
  --keyring "scalekit-kms-keyring" \
  --location "global" \
  --project "YOUR-GCP-PROJECT"
```

The output should list the Scalekit service account with both `roles/cloudkms.cryptoKeyEncrypterDecrypter` and `roles/cloudkms.viewer`. If the bindings are missing, repeat [step 3](#step-3-grant-scalekit-access-to-the-key).

</details>

<details>
<summary>Re-encryption reports unrecoverable rows</summary>

Unrecoverable rows are records that could not be re-encrypted, typically because the previous key version was disabled or destroyed before re-encryption completed. These rows remain encrypted under the prior key version.

Contact [Scalekit support](https://scalekit.com/contact) if you need assistance recovering affected records.

</details>

<details>
<summary>Key resource name is not accepted</summary>

Verify the format is exactly:

```
projects/PROJECT_ID/locations/LOCATION/keyRings/KEYRING_NAME/cryptoKeys/KEY_NAME
```

Run `gcloud kms keys describe` with `--format="value(name)"` to retrieve the exact string. Do not construct it manually.

</details>
