Scalekit enables apps to let users sign in using Twitter (X) as their social connector. This guide walks you through setting up the OAuth 2.0 connection between Scalekit and Twitter, and implementing "Sign in with Twitter" using the Scalekit SDK.

<figure>
  ![A diagram showing "Your Application" connecting to "Scalekit" via OpenID Connect, which links to Twitter (X) using OAuth 2.0.](@/assets/docs/common/poster-scalekit-social.png)
</figure>

By the end of this guide, you will be able to:

1. Set up an OAuth 2.0 connection between Scalekit and Twitter (X)
2. Configure the Twitter Developer Portal with the correct redirect URI and scopes
3. Test the sign-in flow end-to-end from the Scalekit dashboard

## Set up Twitter (X) connection

### Access social login settings

Open your [Scalekit dashboard](https://app.scalekit.com) and navigate to **Authentication** → **Social Login**.

<figure>
  ![Scalekit dashboard showing Social Login under the Authentication section.](@/assets/docs/common/social-connections/1-navigate-to-social-logins.png)
</figure>

### Add Twitter (X) connection

Click **+ Add Connection** and select **Twitter (X)** from the list of available providers.

<figure>
  ![Add social login connections list showing Twitter (X) as an option.](@/assets/docs/common/social-connections/2-list-social-logins.png)
  <figcaption>Add social login connection: Twitter (X)</figcaption>
</figure>

## Configure OAuth settings

The OAuth Configuration page shows three values you need for the Twitter Developer Portal setup:

- **Redirect URI** — Copy this URL. You'll register it with Twitter as the callback URL. Format: `https://{your-subdomain}.scalekit.dev` (development) or `https://{your-subdomain}.scalekit.com` (production).
- **Client ID** — Provided by Twitter after you create an OAuth app.
- **Client Secret** — Provided by Twitter alongside the Client ID.

<figure>
  ![Scalekit OAuth Configuration panel for Twitter (X) showing Redirect URI, Client ID, and Client Secret fields.](@/assets/docs/guides/social-connections/twitter-1.png)
  <figcaption>Twitter (X) OAuth configuration in Scalekit — copy the Redirect URI before opening Twitter Developer Portal.</figcaption>
</figure>

<Aside type="caution" title="Never hard-code credentials">
  Store your Client ID and Client Secret in environment variables. Never commit them to source control or expose them in client-side code.
</Aside>

## Get Twitter OAuth 2.0 credentials

Twitter uses **OAuth 2.0 Authorization Code with PKCE**. You'll create an app in the Twitter Developer Portal to obtain the Client ID and Client Secret.

<Steps>
1. ### Create or open a Twitter Developer account

   Go to the <a href="https://developer.twitter.com/en/portal/dashboard" target="_blank" rel="noopener">Twitter Developer Portal</a> and sign in with your Twitter account.

   If you don't have a developer account yet, apply for access. Basic (free) tier is sufficient for social login.

   <figure>
     ![Twitter Developer Portal dashboard showing Projects & Apps.](@/assets/docs/guides/social-connections/twitter-2.png)
   </figure>

2. ### Create a project and app

   - Click **+ Add App** (or **+ Create Project** if you want to group apps).
   - Select **Production** environment.
   - Give your app a name that identifies your product (e.g., `My App - Social Login`).

   <Aside type="note" title="One app per environment">
     Create separate apps for development and production so each has its own redirect URI and credentials.
   </Aside>

3. ### Configure user authentication settings

   In your app's dashboard, find **User authentication settings** and click **Set up** (or **Edit**).

   Configure the following fields:

   | Setting | Value |
   |---|---|
   | **App permissions** | **Read** — grants access to user profile and email. Select **Read and Write** only if your app needs to post tweets on behalf of users. |
   | **Type of App** | **Web App, Automated App or Bot** — required for server-side OAuth 2.0. |
   | **Callback URI / Redirect URL** | Paste the **Redirect URI** copied from Scalekit. |
   | **Website URL** | Your application's public homepage URL. Required by Twitter. |

   Click **Save** to apply the settings.

4. ### Copy your OAuth 2.0 credentials

   After saving, Twitter displays your **OAuth 2.0 Client ID** and **Client Secret** (also called an OAuth 2.0 Bearer Token Secret).

   - Copy the **Client ID**.
   - Copy the **Client Secret**. Store it immediately — Twitter masks it after you leave the page.

   <Aside type="caution" title="Client secret is shown once">
     If you close the page without saving the Client Secret, you must regenerate it. Regenerating invalidates any existing tokens your users have authorized.
   </Aside>

5. ### Enter credentials in Scalekit

   Return to the Scalekit dashboard OAuth Configuration page for Twitter (X):

   - Paste your **Client ID**
   - Paste your **Client Secret**
   - Click **Save Changes**

   <figure>
     ![Scalekit dashboard showing Twitter credentials filled in with Client ID, Client Secret, and Redirect URI.](@/assets/docs/common/social-connections/4-after-oauth-config.png)
   </figure>
</Steps>

## OAuth 2.0 scopes

Scalekit requests the following Twitter OAuth 2.0 scopes by default:

| Scope | Purpose |
|---|---|
| `tweet.read` | Read tweets on behalf of the user |
| `users.read` | Read user profile information (name, username, profile picture) |
| `offline.access` | Issue a refresh token so sessions can be renewed without re-authorization |

<Aside type="note" title="Email address availability">
  Twitter's standard OAuth 2.0 scopes do not include an email scope. If your application requires the user's email address, prompt users to provide it separately after sign-in. Twitter only exposes email via the v1.1 API with a verified phone number on the account.
</Aside>

## Login prompt behavior

Scalekit lets you control how Twitter presents the authentication screen to returning users:

- **Auto sign-in (default)** — Automatically completes sign-in if the user has an active Twitter session and has previously authorized your app. No confirmation screen is shown.

- **Consent** — Forces the authorization server to show the consent screen even if the user has already granted permissions. Useful when you add new scopes.

- **None** — Performs a background authentication check. Returns an error if the user is not already authenticated or has not authorized your app. Use this to silently check whether a user session exists.

<Aside type="tip" title="Twitter app permissions change">
  If you update your app's permissions in the Twitter Developer Portal (for example, adding Direct Messages access), existing authorized users must re-authorize. Set the prompt to **Consent** after a permissions change to ensure users see the updated scope request.
</Aside>

Click **Test Connection** in the Scalekit dashboard. You'll be redirected to the Twitter authorization screen. After approving access, a summary table shows the user profile information that will be passed to your application.

## Troubleshooting

<details>
<summary>Callback URL not approved / redirect_uri mismatch error</summary>

The redirect URI in Scalekit must exactly match what you entered in Twitter's **Callback URI / Redirect URL** field — including the protocol (`https://`), subdomain, and path. Check for trailing slashes or typos.

</details>

<details>
<summary>Client Secret was not saved</summary>

If you did not copy the Client Secret before leaving the Twitter Developer Portal, navigate to your app → **Keys and tokens** → **OAuth 2.0 Client ID and Client Secret** → **Regenerate**. Update the new secret in Scalekit.

</details>

<details>
<summary>App permissions: "Read" vs "Read and Write"</summary>

For social login (sign-in only), **Read** permissions are sufficient. Selecting **Read and Write** expands the permission consent screen shown to users and may increase friction. Only request write permissions if your app actively posts content to Twitter on the user's behalf.

</details>

<details>
<summary>Twitter only allows one active redirect URI per app</summary>

Twitter supports multiple callback URIs. If you need both a development (`*.scalekit.dev`) and a production (`*.scalekit.com`) redirect URI, add both in the **Callback URI / Redirect URL** field, each on a new line.

</details>
