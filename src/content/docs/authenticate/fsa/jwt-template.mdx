Every access token Scalekit issues already carries a working set of claims: `sub`, `roles`, `permissions`, the organization the user signed into, and identifiers like `external_id`. That covers what most B2B applications need to authenticate a request.

It doesn't cover everything a request might need to decide. A detail on the user's profile, something in their organization's metadata, a field you've stored on the membership itself: none of that shows up in the token by default. Add it as a custom claim instead of reaching for your database on every request, and it's already there once your API finishes verifying the token.

## Fields you can reference

Every expression reads from one of two objects: `user` or `organization`.

**User fields:**

| Field | Holds |
|---|---|
| `user.id` | Scalekit user ID |
| `user.email`, `user.email_verified` | Primary email and its verification state |
| `user.name`, `user.given_name`, `user.family_name` | Display, first, and last name |
| `user.external_id` | Your system's identifier for this user |
| `user.metadata` | Key-value data you've attached to the user |
| `user.custom_attributes` | Custom profile attributes |
| `user.locale` | Language and region preference |
| `user.current_membership.roles` | Roles the user holds in the organization they're signing into |
| `user.current_membership.status` | Membership status, for example `active` |
| `user.current_membership.metadata` | Key-value data on the membership itself |

**Organization fields:**

| Field | Holds |
|---|---|
| `organization.id` | Scalekit organization ID |
| `organization.name` | Organization display name |
| `organization.external_id` | Your system's identifier for this organization |
| `organization.metadata` | Key-value data you've attached to the organization |

## Add a custom claim

<Steps>
1. ### Open Token Claims

   Go to **Dashboard > Customize > Token Claims**. The page lists your **default claims** (the ones Scalekit always includes) and, below them, any **custom claims** you've added.

2. ### Define the claim

   Click **Add custom claim** and fill in two fields:

   - **Claim key**: the name it appears under in the token, for example `a_user_email`.
   - **Expression**: where the value comes from, for example `user.email`.

   ![Add custom claim panel with a claim key field and an expression field.](@/assets/docs/jwt-template/add-custom-claim.png)

   You don't need any special syntax here, just the dot-path to the field you want, like `user.email` or `organization.metadata.plan`. Scalekit evaluates it for every token going forward.

3. ### Save it

   Scalekit applies the claim immediately; there's no separate publish or enable step. Use the pencil icon next to a claim to edit its expression, or the trash icon to remove it.
</Steps>

## Preview before you rely on it

The **Token preview** panel on the right of the Token Claims page resolves every claim, default and custom, against a real person. Search for a user, and the JSON view updates with the exact payload that user would get in their next token.

![Token Claims page with a resolved JWT payload in the preview panel.](@/assets/docs/jwt-template/token-claims-page.png)

For a true end-to-end check, sign in through your OIDC flow and decode the access token you receive. That confirms the claim survives the full login path, not just the preview.

<Aside type="note" title="A claim can resolve to nothing">
If an expression's field is missing for a given user (no `metadata.department` set, for example), Scalekit leaves that claim out of the token rather than adding an empty one. A login never fails because of a claim expression; worst case, the claim just isn't there. Preview against a few different users to catch this before you depend on the claim in production.
</Aside>

## Write the expression

An expression is a dot-path into `user` or `organization`, evaluated with [expr-lang](https://expr-lang.org/docs/language-definition):

| What you want | Expression |
|---|---|
| A profile field | `user.email` |
| Something from your own metadata | `organization.metadata.plan` |
| The first role a user holds | `user.current_membership.roles[0]` |
| A default when a field might be unset | `organization.metadata.plan ?? "free"` |
| A yes/no from a membership check | `"admin" in user.current_membership.roles` |

expr-lang also has string and array helpers (`split`, `trim`, `len`, `filter`, and more) for anything more involved than a plain lookup.

## Claims you can't override

`sub`, `iss`, `aud`, `exp`, `iat`, `nbf`, `jti`, `oid`, `sid`, `xoid`, `xuid`, `roles`, and `permissions` are already set by Scalekit on every token. A custom claim can't reuse one of these keys, so pick a name that doesn't collide, the way `a_user_email` avoids colliding with any default claim.
