Test users let you run Playwright, Cypress, and other E2E tests against Scalekit-hosted login flows without waiting for real emails. Instead of sending a verification email, Scalekit accepts a static 6-digit code you configure — so your tests are fast, reliable, and inbox-free.

Test Users is an allowlist on top of real user records — it does not create users for you. Add the user to an organization first, then add the same email to the allowlist.

**What doesn't change:** Test Users only affects OTP email delivery. Everything else — organization selection, session creation, audit logs, webhooks, and post-login redirects — works exactly the same as for a real user.

## Email format

Every allowlisted email must contain `+sktest` in the local part (before `@`). The domain doesn't matter — use any domain you control.

| Email | Valid? | Reason |
| --- | :---: | --- |
| `alice+sktest@acmecorp.com` | ✓ | `+sktest` present in the local part |
| `bob+sktest@anything.example` | ✓ | Any domain works |
| `alice@acmecorp.com` | ✗ | Missing `+sktest` marker |
| `alice+test@acmecorp.com` | ✗ | Marker must be exactly `+sktest` |
| `sktest+alice@acmecorp.com` | ✗ | Must appear after `+`, not before `@` |

## Prerequisites

- Hosted auth screens enabled in your Scalekit environment.
- Passwordless mode set to `OTP` or `LINK_OTP` — `LINK`-only mode is not supported (see [Limitations](#limitations)).
- Admin access to **Environment settings** in the dashboard.

On every login attempt, Scalekit checks for the `+sktest` marker before anything else — addresses without it skip the allowlist entirely and follow the normal flow.

## Set up a test user

<Steps>
1. ## Add the user to an organization

   Every test user must belong to an organization. In the dashboard, go to **Organizations**, open the org you want to use (or create one — e.g. `e2e-tests`), click the **Users** tab, and add a new user with a `+sktest` email.

   The `+sktest` marker is required in the local part of the address (before `@`). For example: `alice+sktest@acmecorp.com`.

2. ## Open Test users settings

   Go to **Dashboard > Environment settings > Test users**.

   ![Test users settings page](@/assets/docs/authenticate/test-users/test-users-settings.png)

   <Aside type="note" title="Default state per environment">
     Dev environments have Test Users enabled by default. Production environments have it disabled. A warning banner appears in the dashboard whenever Test Users is on in a production environment.
   </Aside>

3. ## Enable and add the email to the allowlist

   Toggle **Enable test users**, then add the same `+sktest` email from step 1.

   ![Test users allowlist with an email added](@/assets/docs/authenticate/test-users/test-users-allowlist.png)

   You can add up to 5 emails per environment. Emails are saved in lowercase and duplicates are removed automatically.

4. ## Set the static confirmation code

   Enter a **Static confirmation code** — exactly 6 digits. The default is `424242`. This is the code your test will type at the OTP prompt.

5. ## Save

   Click **Save**. Reload the page to confirm the settings persisted.
</Steps>

## Run an E2E test

The examples below test a full OTP login for `alice+sktest@acmecorp.com`. Replace the URL and selectors to match your hosted auth screen.

<Aside type="tip" title="Use environment variables">
Store the test email and static code as environment variables (`SCALEKIT_TEST_USER_EMAIL`, `SCALEKIT_TEST_USER_CODE`) — don't hard-code them in test files.
</Aside>

<Tabs syncKey="e2e-framework">
  <TabItem label="Playwright">
```ts
import { test, expect } from '@playwright/test'

const TEST_USER_EMAIL = process.env.SCALEKIT_TEST_USER_EMAIL!
const TEST_USER_CODE  = process.env.SCALEKIT_TEST_USER_CODE!

test('OTP login with test user', async ({ page }) => {
  await page.goto('https://app.example.com/login')

  await page.getByLabel('Email').fill(TEST_USER_EMAIL)
  await page.getByRole('button', { name: 'Continue' }).click()

  // No real email sent — type the static code
  await page.getByLabel('Verification code').fill(TEST_USER_CODE)
  await page.getByRole('button', { name: 'Verify' }).click()

  await expect(page).toHaveURL('https://app.example.com/dashboard')
})
```
  </TabItem>
  <TabItem label="Cypress">
```ts
const TEST_USER_EMAIL = Cypress.env('SCALEKIT_TEST_USER_EMAIL')
const TEST_USER_CODE  = Cypress.env('SCALEKIT_TEST_USER_CODE')

describe('OTP login with test user', () => {
  it('logs in using the static code', () => {
    cy.visit('https://app.example.com/login')

    cy.get('input[type="email"]').type(TEST_USER_EMAIL)
    cy.get('button').contains('Continue').click()

    // No real email sent — type the static code
    cy.get('input[name="otp"]').type(TEST_USER_CODE)
    cy.get('button').contains('Verify').click()

    cy.url().should('include', '/dashboard')
  })
})
```
  </TabItem>
</Tabs>

## Verify the flow

After running the test, check:

- **No email arrived** in the inbox for the test address.
- **Auth logs** in **Dashboard > Auth logs** show two test-user events:
  - `auth.test_user_verification_issued` — shown as *Verification email suppressed (test user)*
  - `auth.test_user_otp_verified` — shown as *OTP verification successful (test user)*

  ![Auth log showing test-user events](@/assets/docs/authenticate/test-users/test-users-auth-log.png)

  See [Auth logs](/guides/dashboard/auth-logs) for how to filter these entries.
- **User and session exist normally** — the test user's session and org membership appear in the dashboard just like a real user's.

## Limitations

- **`LINK`-only mode is not supported.** If your environment uses `LINK`-only passwordless, Scalekit sends the normal magic-link email even if the address is on the allowlist. Switch to `OTP` or `LINK_OTP`.
- **`LINK_OTP` suppresses the email, but the static code still works.** The magic link is never sent, but entering the 6-digit code completes verification.
- **Maximum 5 emails per environment.** Use different `+sktest` addresses for parallel test runs.
- **Signup gating is not bypassed.** If signup is disabled in your environment, the test user must already exist as an org member before login works. See [Testing login when signup is disabled](#testing-login-when-signup-is-disabled).
- **SSO, social login, magic-link signup, and org invites are unaffected.** Test Users only applies to hosted OTP flows.

## Production safety

<Aside type="caution" title="Test Users bypass real email verification">
Keep Test Users off in production unless you have a deliberate reason to enable it. When it is on, a warning banner appears in the dashboard — leave it visible.

Every allowlisted email must contain `+sktest`. This makes test identities easy to spot in logs and audit records.
</Aside>

## Common scenarios

<details>
<summary>Why am I still getting a real verification email?</summary>

All three conditions must be true for Test Users to apply:

- Test Users is enabled in the environment.
- The exact email is on the allowlist (case-insensitive match).
- The email contains `+sktest` in its local part.

If `+sktest` is missing, Scalekit skips the allowlist entirely and sends the normal email — no database lookup occurs.

Also check that your passwordless mode is `OTP` or `LINK_OTP`, not `LINK`-only.

</details>

<details>
<summary>Can I use a different code for each test run?</summary>

No — the static code is set per environment. For parallel isolation, add multiple `+sktest` addresses (up to 5) and assign a different one to each test run.

</details>

<details>
<summary>How do I keep Test Users out of production CI?</summary>

Never enable Test Users in your production environment. In CI, point to a dev or staging environment. You can also guard your test runner with an environment check — for example, verify that `SCALEKIT_ENVIRONMENT_URL` contains `.dev.` or `.staging.` before running E2E tests.

</details>

<details id="testing-login-when-signup-is-disabled">
<summary>Testing login when signup is disabled in my environment</summary>

Test Users does not bypass signup gating. If signup is off, the `+sktest` user must already exist inside an organization before login will work.

Use this one-time setup in your test fixture:

1. **Create or pick an organization** — e.g. `e2e-tests` — via the dashboard or `POST /api/v1/organizations`.

2. **Create the user inside that organization:**
