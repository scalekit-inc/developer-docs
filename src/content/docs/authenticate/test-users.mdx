Test users let you run Playwright, Cypress, and other E2E tests against login and signup flows without waiting for real emails for OTP or magic links. Instead of sending a verification email, Scalekit accepts a static 6-digit verification code you configure so that your tests are fast, reliable, and inbox-free.

**What doesn't change:** Test Users only affects OTP email delivery. Everything else like organization creation, user creation, session creation, audit logs, webhooks, and post-login redirects, works exactly the same as for a real user.

## Prerequisites

- Ensure that **Magic Link & OTP** authentication mode is enabled and the method is set to `Verification Code` or `Magic Link + Verification Code`. `Magic Link` mode is not supported.

## Configure test users

Test users allowlist is set per environment upto 5 emails. Configure it once; every signup and login test in that environment uses the same emails and static verification code.

<Steps>
1. ## Open Test users settings

   Go to **Dashboard > Environment settings > Test users**.

   ![Test users settings page](@/assets/docs/authenticate/test-users/test-users-settings.png)

2. ## Enable and add emails to the allowlist

   **Enable test users**, then add up to 5 test emails. Emails are stored lowercase and de-duplicated. Every allowlisted email must contain `+sktest` in the local part (before `@`). The domain doesn't matter.

   | Email | Valid? | Reason |
   | --- | :---: | --- |
   | `john+sktest@acmecorp.com` | ✓ | `+sktest` present in the local part |
   | `bob+sktest@anything.example` | ✓ | Any domain works |
   | `john@acmecorp.com` | ✗ | Missing `+sktest` marker |
   | `john+test@acmecorp.com` | ✗ | Marker must be exactly `+sktest` |
   | `sktest+john@acmecorp.com` | ✗ | Must appear after `+`, not before `@` |

   ![Test users allowlist with an email added](@/assets/docs/authenticate/test-users/test-users-allowlist.png)

3. ## Set the static confirmation code

   Enter a **Static confirmation code** — exactly 6 digits. The default is `424242`. This is the code your tests will type at the OTP prompt.

4. ## Save

   Click **Save**. Reload the page to confirm the settings persisted.
</Steps>

## Test the signup flow

Use this when you want to verify that a brand new user can complete signup and land in an organization. At the end of this flow, the test user exists as a member of a new organization. 


<Aside type="tip" title="Use environment variables">
Store the test email, static code, and authorization URL as environment variables (`SCALEKIT_TEST_USER_EMAIL`, `SCALEKIT_TEST_USER_CODE`, `SCALEKIT_TEST_AUTH_URL`) — don't hard-code them in test files.
</Aside>

Drive the test through Scalekit's authorization endpoint with `prompt=create` so it lands on the hosted signup screen — that way the test is independent of your app's own routing:

```sh title="Authorization URL (signup)" showLineNumbers=false frame
<SCALEKIT_ENVIRONMENT_URL>/oauth/authorize?
  response_type=code&
  client_id=<SCALEKIT_CLIENT_ID>&
  redirect_uri=<CALLBACK_URL>&
  scope=openid+profile+email+offline_access&
  state=<RANDOM_STATE>&
  prompt=create
```

After OTP verification, Scalekit redirects to `<CALLBACK_URL>` with an authorization code; assert against that URL in your test.

<Tabs syncKey="e2e-framework">
  <TabItem label="Playwright">
```ts
import { test, expect } from '@playwright/test'

const AUTH_URL        = process.env.SCALEKIT_TEST_AUTH_URL!  // includes &prompt=create
const TEST_USER_EMAIL = process.env.SCALEKIT_TEST_USER_EMAIL!
const TEST_USER_CODE  = process.env.SCALEKIT_TEST_USER_CODE!

test('OTP signup with test user', async ({ page }) => {
  // drives the browser directly to Scalekit's hosted screen via the authorize URL
  await page.goto(AUTH_URL)

  await page.getByLabel('Email').fill(TEST_USER_EMAIL)
  await page.getByRole('button', { name: 'Continue' }).click()

  // No real email sent — type the static code
  await page.getByLabel('Verification code').fill(TEST_USER_CODE)
  await page.getByRole('button', { name: 'Verify' }).click()

  await expect(page).toHaveURL(/\/dashboard/)
})
```
  </TabItem>
  <TabItem label="Cypress">
```ts
const AUTH_URL        = Cypress.env('SCALEKIT_TEST_AUTH_URL')  // includes &prompt=create
const TEST_USER_EMAIL = Cypress.env('SCALEKIT_TEST_USER_EMAIL')
const TEST_USER_CODE  = Cypress.env('SCALEKIT_TEST_USER_CODE')

describe('OTP signup with test user', () => {
  it('signs up using the static code', () => {
    // drives the browser directly to Scalekit's hosted screen via the authorize URL
    cy.visit(AUTH_URL)

    cy.get('input[type="email"]').type(TEST_USER_EMAIL)
    cy.get('button').contains('Continue').click()

    // No real email sent — type the static code
    cy.get('input[name="otp"]').type(TEST_USER_CODE)
    cy.get('button').contains('Verify').click()

    cy.url().should('include', '/dashboard')
  })
})
```
  </TabItem>
</Tabs>

**Verify:** After running the test, the new user appears in **Dashboard > Organizations > \{org\} > Users**. Auth logs in **Dashboard > Auth logs** show:
- `auth.test_user_verification_issued` — *Verification email suppressed (test user)*
- `auth.test_user_otp_verified` — *OTP verification successful (test user)*

## Test the login flow

Use this when you want to verify that an existing `+sktest` user can sign back in.

The test user must already exist as a member of an organization before the test runs. Test Users does not bypass signup gating — if signup is disabled, or you want deterministic fixture data, pre-create the user via the dashboard or SDK first.

### Prerequisite: ensure the user exists in an organization

You can add the user manually in **Dashboard > Organizations > \{org\} > Users**, or create them from your test fixture:


## Production safety

<Aside type="caution" title="Test Users bypass real email verification">
Keep Test Users off in production unless you have a deliberate reason to enable it. When it is on, a warning banner appears in the dashboard — leave it visible.

Every allowlisted email must contain `+sktest`. This makes test identities easy to spot in logs and audit records.
</Aside>

## Common scenarios

<details>
<summary>Why am I still getting a real verification email?</summary>

All three conditions must be true for Test Users to apply:

- Test Users is enabled in the environment.
- The exact email is on the allowlist (case-insensitive match).
- The email contains `+sktest` in its local part.

If `+sktest` is missing, Scalekit skips the allowlist entirely and sends the normal email — no database lookup occurs.

Also check that your passwordless mode is `OTP` or `LINK_OTP`, not `LINK`-only.

</details>

<details>
<summary>Can I use a different code for each test run?</summary>

No — the static code is set per environment. For parallel isolation, add multiple `+sktest` addresses (up to 5) and assign a different one to each test run.

</details>

<details>
<summary>How do I keep Test Users out of production CI?</summary>

Never enable Test Users in your production environment. In CI, point to a dev or staging environment. You can also guard your test runner with an environment check — for example, verify that `SCALEKIT_ENVIRONMENT_URL` contains `.dev.` or `.staging.` before running E2E tests.

</details>
