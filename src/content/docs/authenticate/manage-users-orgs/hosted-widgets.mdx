**Hosted widgets** are Scalekit-hosted full pages that give your SaaSKit customers a branded self-serve experience for organization admin, member management, enterprise setup, and personal account settings. Redirect authenticated users to `<SCALEKIT_ENVIRONMENT_URL>/ui/`. Scalekit enforces permissions and feature entitlements, applies your [application branding](/fsa/guides/login-page-branding/), and can use your [custom domain](/guides/custom-domain/).

Hosted widgets use the **Scalekit platform session** for the signed-in user. They are available for **SaaSKit** integrations that rely on that session model.

```sh title="Hosted widgets portal URL" showLineNumbers=false frame
<SCALEKIT_ENVIRONMENT_URL>/ui/  # https://your-app-env.scalekit.com/ui/
```

## How Hosted widgets work

Hosted widgets are **full pages on Scalekit's origin**, not React components you embed in your DOM. Your app redirects the browser to Scalekit. Scalekit renders the UI and talks to Scalekit on behalf of the current user. You do not send application data into the widgets over webhooks to hydrate the UI.

Key properties:

- **SaaSKit session model**: The portal expects a Scalekit platform session for the end user. Follow the [SaaSKit quickstart](/authenticate/fsa/quickstart/) and [session management](/authenticate/fsa/manage-session/) guides for how your app establishes and stores sessions.
- **Single portal entry**: You send users to `/ui/`. You do not configure deep links into individual modules. After the portal loads, users navigate within Hosted widgets to the screens they can access.
- **Access control**: Scalekit shows each module based on the user's permissions and the organization's feature entitlements.
- **Branding**: Logo, favicon, primary color, and related options come from your shared **Customize** settings. You get a consistent look across hosted experiences, not per-component React theming.
- **Embed**: The self-serve portal at `/ui/` is a **redirect** experience. The only iframe-oriented hosted surface is the [Admin portal](/guides/admin-portal/) (shareable link or embed) for enterprise SSO and SCIM setup.

<Aside type="note" title="Embeddable widgets">
  Embeddable, in-app widgets (for example, component-style embeds similar to some third-party auth UIs) are on the roadmap. Contact Scalekit if you need guidance for a specific integration shape.
</Aside>

## Choose Hosted widgets or Admin portal

| Need | Use |
| --- | --- |
| Logged-in SaaSKit users self-serve members, profiles, security, domains, session policy, SSO, and SCIM in one branded portal | **Hosted widgets** (`/ui/`) |
| Share a magic link with enterprise IT, or **embed** SSO and SCIM setup in an iframe | **[Admin portal](/guides/admin-portal/)** |
| Fully custom admin UI without Scalekit-hosted pages | Not supported for SaaSKit today. SaaSKit self-serve admin depends on the Scalekit platform session and Hosted widgets (or Admin portal for shareable/embed SSO and SCIM). |

Admin portal remains documented under its own name because it supports **shareable links** and **iframe embed**. Product-wise it is part of the broader hosted self-serve surface area Scalekit is expanding.

## When to use Hosted widgets

Use Hosted widgets when you want SaaSKit customers to manage organizations and accounts without you shipping equivalent admin UIs.

- **Advantages**: Customers self-serve member management, single sign-on (SSO), SCIM provisioning, domain verification, session policy, and personal profile/security. Permissions and feature entitlements are enforced automatically. The portal picks up new Scalekit capabilities as they ship.
- **Trade-offs**: You redirect users out of your app into Scalekit-hosted pages. Branding follows shared Customize settings, not component-level control. You do not deep-link to a specific module from your app.

## Integrate the portal

1. Complete SaaSKit authentication so the user has a Scalekit platform session. See the [quickstart](/authenticate/fsa/quickstart/).
2. Add an entry point in your product (for example, **Settings** or **Organization**).
3. Redirect the browser to `<SCALEKIT_ENVIRONMENT_URL>/ui/` (or your custom domain equivalent).
4. Users move between organization and user modules inside the portal using its navigation.

Scalekit checks the user's roles and permissions and shows only the modules they can use. If you store access and refresh tokens in your own backend cookies, follow the session patterns in the [SaaSKit quickstart](/authenticate/fsa/quickstart/) and [manage sessions](/authenticate/fsa/manage-session/) guides so the browser session Scalekit expects stays consistent with your app.

## Widget catalog

| Module | Audience | What customers do | Gated by |
| --- | --- | --- | --- |
| Organization settings | Org admins / permitted roles | View and update organization profile details | `sk_org_settings_*` permissions |
| Member management | Org admins / permitted roles | View members, invite users, change roles, remove members | `sk_org_users_*` permissions |
| SSO configuration | Org admins / permitted roles | Connect and manage enterprise SSO | Permissions + **SSO** feature entitlement |
| SCIM configuration | Org admins / permitted roles | Set up directory sync (SCIM) | Permissions + **SCIM** feature entitlement |
| Domain verification | Org admins / permitted roles | Add domains and publish DNS TXT records for verification | Permissions + **Domain Verification** entitlement |
| Session policy | Org admins / permitted roles | Set organization absolute and idle session timeouts | Permissions + **Session Policy** entitlement |
| User profile | Any authenticated user | Manage personal profile details | Authenticated user (no org entitlement) |
| User security | Any authenticated user | Manage passkeys, view and revoke sessions | Authenticated user (no org entitlement) |
| Login / signup screens | Unauthenticated users | Complete hosted authentication | Authorization URL `prompt` (see below) |

### Organization modules

Organization modules appear only when the user has the required permissions and, where noted, the organization has the feature enabled.

#### Organization settings

Customers view and manage the organization profile, including allowed email domains, under **Organization settings**.

![](@/assets/docs/hosted-widgets/org_settings.png)

#### Member management

Customers view members, invite users, manage roles, and remove members.

![](@/assets/docs/hosted-widgets/org_member.png)

#### SSO configuration

Customers set up and manage SSO with identity-provider-specific guidance in the portal.

Enable SSO for the organization in the Scalekit dashboard or with the [SDK](/authenticate/auth-methods/enterprise-sso/#enable-sso-for-the-organization). The module appears only when SSO is entitled for the organization.

![](@/assets/docs/hosted-widgets/org_sso.png)

#### SCIM configuration

Customers set up SCIM provisioning with identity-provider-specific guidance.

Enable SCIM in the Scalekit dashboard or with the [SDK](/guides/user-management/scim-provisioning/#enable-scim-provisioning-for-the-organization). The module appears only when SCIM is entitled for the organization.

![](@/assets/docs/hosted-widgets/org_scim.png)

#### Domain verification

Customers add and verify domains they own, which supports Home Realm Discovery and SCIM for the organization. See [Organization domains](/authenticate/manage-users-orgs/organization-domains/).

After a customer enters a domain, the portal shows the DNS TXT record to publish. Scalekit verifies ownership in the background and marks the domain verified when the record propagates.

Enable **Domain Verification** for the organization in the Scalekit dashboard or with the [organization settings API](/apis/#tag/organizations/PATCH/api/v1/organizations/{id}/settings).

![Domain verification via hosted widgets](@/assets/docs/organization-domains/2026-05-26-19-06-40.png)

#### Session policy

Customers set organization session policy (absolute and idle timeouts) that override application defaults. Scalekit enforces the stricter of the two values.

Enable **Session Policy** for the organization in the Scalekit dashboard or with the [organization settings API](/apis/#tag/organizations/PATCH/api/v1/organizations/{id}/settings).

![](@/assets/docs/hosted-widgets/2026-05-21-17-12-14.png)

### User modules

User modules are available to authenticated users. They are not gated by organization feature entitlements or organization-level Scalekit permissions.

#### User profile

Customers manage name, email, and other personal account details.

![](@/assets/docs/hosted-widgets/user_profile.png)

#### User security

Customers register and manage passkeys, view active sessions, and revoke sessions.

![](@/assets/docs/hosted-widgets/user_security.png)

## Hosted login and signup screens

Login and signup are **hosted authentication pages**, separate from the `/ui/` portal modules. They do not change which organization or user modules appear after sign-in. Use them when you want Scalekit to render branded auth screens instead of building those screens yourself.

Pass `prompt` on the authorization URL:

<Tabs>
  <TabItem value="login" label="Login">
```sh title="Authorization URL (login)" showLineNumbers=false frame
<SCALEKIT_ENVIRONMENT_URL>/oauth/authorize?
  response_type=code&
  client_id=<SCALEKIT_CLIENT_ID>&
  redirect_uri=<CALLBACK_URL>&
  scope=openid+profile+email+offline_access&
  state=<RANDOM_STATE>&
  prompt=login
```

Pass `prompt=login` to show the login page. Users land on `<SCALEKIT_ENVIRONMENT_URL>/a/auth/login`.

![Login page of coffee desk app](@/assets/docs/hosted-widgets/login.png)
  </TabItem>

  <TabItem value="signup" label="Signup">

  response_type=code&

  scope=openid+profile+email+offline_access&

## Access management

Hosted widgets enforce access with **Scalekit permissions**. Map these permissions to application roles. When a user opens the portal, Scalekit checks permissions and shows the modules they can use.

| Permission | Purpose |
| --- | --- |
| `sk_org_settings_read` | View organization profile and settings |
| `sk_org_settings_manage` | View and modify organization profile and settings |
| `sk_org_users_read` | View users in an organization |
| `sk_org_users_invite` | Invite new users to an organization |
| `sk_org_users_delete` | Remove users from an organization |
| `sk_org_users_role_change` | Change roles of users in an organization |
| `sk_org_sso_read` | View SSO configuration for an organization |
| `sk_org_sso_manage` | View and modify SSO configuration for an organization |
| `sk_org_scim_read` | View SCIM configuration for an organization |
| `sk_org_scim_manage` | View and modify SCIM configuration for an organization |
| `sk_org_session_policy_read` | View session policy for an organization |
| `sk_org_session_policy_manage` | View and manage session policy for an organization |

<Aside type="note" title="Default Admin and Member roles">
  Scalekit creates **Admin** and **Member** roles for every environment by default and maps Scalekit permissions to those roles. The Admin role includes all Scalekit permissions and can access all organization modules the org is entitled to. The Member role has limited organization access (view organization settings and organization members). Both roles can use user modules (profile and security).

  Customize role mappings or create a [custom role](/authenticate/authz/create-roles-permissions/) and assign Scalekit permissions to control portal access.
</Aside>

## Branding and customization

Hosted widgets use the same application branding as your other hosted experiences: logo, favicon, primary color, and related **Customize** settings. See [Login page branding](/fsa/guides/login-page-branding/) and [Organization branding](/fsa/guides/organization-branding/).

Serve Hosted widgets on your own hostname with a [custom domain](/guides/custom-domain/).

## Common scenarios

<details>
<summary>What happens if a user does not have a session?</summary>

Hosted widgets expect a Scalekit platform session. If no session exists, Scalekit redirects the user so your application can start authentication again (typically via your configured login endpoint). Prefer sending only authenticated users to `/ui/`. Establish sessions with the [SaaSKit quickstart](/authenticate/fsa/quickstart/) flow rather than assuming a custom session-only backend can open the portal without a Scalekit user session.

</details>

<details>
<summary>What happens when a user logs out from Hosted widgets?</summary>

When a user logs out from Hosted widgets, Scalekit sends them to your configured post-logout destination. Your app session and the Scalekit session can fall out of sync if you also store tokens or sessions in your own backend.

If you manage tokens in your backend (see the [SaaSKit quickstart](/authenticate/fsa/quickstart/) and [session management](/authenticate/fsa/manage-session/)), propagate logout yourself with one of these approaches:

- [Back-channel logout](/guides/dashboard/redirects/#back-channel-logout-url) so Scalekit can notify your app when the session ends
- The [user logout webhook](/apis/#webhook/userlogout) so your backend learns about session termination

</details>

<details>
<summary>Do I need webhooks to supply data to Hosted widgets?</summary>

No. Scalekit hosts the pages and loads the data those pages need. You do not hydrate Hosted widgets from your backend with webhooks. Use webhooks and APIs for **your** application workflows (for example logout propagation or provisioning events), not to feed the Hosted widgets UI.

</details>

<details>
<summary>Can I embed Hosted widgets in an iframe?</summary>

The `/ui/` portal is a redirect-based experience. For iframe embed and shareable magic links aimed at enterprise SSO and SCIM setup, use the [Admin portal](/guides/admin-portal/). Embeddable in-app widgets are on the roadmap; contact Scalekit for your use case.

</details>
