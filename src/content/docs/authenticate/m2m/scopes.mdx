**Scopes** are named permissions attached to an OAuth client and embedded in access tokens. In machine-to-machine (M2M) flows, they tell your API *what* an automated client is allowed to do—separate from *who* the client is (identified by `client_id` / `sub`).

## Why scopes matter

Without scopes, any valid client could call any endpoint. Scopes let you:

- Grant **least privilege** (for example only `read:invoices`, not `delete:invoices`).
- **Document** intended usage for your customers when they register API clients.
- **Enforce** checks in your API layer by reading the `scopes` claim on the JWT.

## How scopes work in Scalekit M2M

1. When you **register an API client** for an organization, you pass a list of scope strings in the `scopes` field (REST) or `scopes` on the client object (SDKs).
2. Scalekit stores those scopes on the client and includes them in issued access tokens.
3. Your API **validates the JWT** (see [JWKS and token verification](/authenticate/m2m/jwks-and-token-validation/)), then checks that the token’s `scopes` include what the route requires.

Scopes are **strings you define** to match your product’s permission model. Use a consistent pattern such as `resource:action` (for example `deployments:read`, `deployments:write`).

## Define scopes when registering a client

Scopes are set at **client creation** (and can be updated when your integration updates the client via the API). Example shape:

```json title="scopes on create client (illustrative)"
"scopes": [
  "deploy:applications",
  "read:deployments"
]
```

The same values appear on the client record and in tokens issued for that client.

<Aside type="note" title="Token response vs JWT payload">
  The `/oauth/token` response may include a space-separated `scope` string for OAuth compatibility. The validated JWT payload carries scopes as a **`scopes` array**—use that for authorization logic in your API. See the [quickstart](/authenticate/m2m/api-auth-quickstart/) for a decoded example.
</Aside>

## Validate scopes on your API

<Steps>

1. **Authenticate** the request by verifying the JWT signature and standard claims ([JWKS and token verification](/authenticate/m2m/jwks-and-token-validation/)).

2. **Read `scopes`** from the verified payload. Scalekit includes granted scopes as an array:

   ```json title="scopes in JWT payload (example)"
   "scopes": [
     "deploy:applications",
     "read:deployments"
   ]
   ```

3. **Compare** against what the route requires—for example require `deploy:applications` on `POST /deploy`. Return `403` if the scope is missing.

4. **Use the Scalekit SDK** where possible: validation helpers can check required scopes together with signature verification (see the [quickstart](/authenticate/m2m/api-auth-quickstart/)).

</Steps>
