Machine-to-machine (M2M) access tokens issued by Scalekit are **JSON Web Tokens (JWTs)** signed with an RSA private key. Your API must verify the signature before trusting any claim in the token. **JSON Web Key Set (JWKS)** is how you obtain the public keys needed for that verification.

## What JWKS is

A JWKS is a JSON document that lists one or more **JWKs** (JSON Web Keys)—public key material identified by a `kid` (key ID). When Scalekit signs a JWT, it puts the matching `kid` in the JWT header. Your validator:

1. Reads `kid` from the token header.
2. Fetches the JWKS from Scalekit (or uses a cached copy).
3. Selects the key with that `kid` and verifies the signature (typically **RS256**).

This pattern avoids shipping public keys in your codebase: you always use the keys your authorization server publishes.

## Scalekit JWKS endpoint

Each environment exposes its signing keys at:

```http
GET https://<SCALEKIT_ENVIRONMENT_URL>/keys
```

Replace `<SCALEKIT_ENVIRONMENT_URL>` with the same base URL you use for `/oauth/token` (for example `https://your-app.scalekit.dev`).

The response follows the standard JWKS shape:

```json title="Example JWKS document"
{
  "keys": [
    {
      "use": "sig",
      "kty": "RSA",
      "kid": "snk_58327480989122566",
      "alg": "RS256",
      "n": "…",
      "e": "AQAB"
    }
  ]
}
```

Use the key where `use` is `sig` and `alg` is `RS256` for access tokens.

## How to verify an access token

<Steps>

1. **Extract the bearer token** from the `Authorization: Bearer <token>` header (or your API’s agreed mechanism).

2. **Decode the JWT header** (base64url, first segment) and read `kid` and `alg`. Do not trust the payload until the signature is verified.

3. **Resolve the signing key** using `kid`:
   - Fetch `https://<SCALEKIT_ENVIRONMENT_URL>/keys`, or
   - Use a JWKS client library (for example `jwks-rsa` in Node.js) pointed at that URL, with **caching** and refresh on unknown `kid`.

4. **Verify the signature** with your JWT library using the public key and the algorithm from the header (RS256 for Scalekit access tokens).

5. **Validate standard claims**, for example:
   - `exp` — token not expired
   - `iss` — issuer matches your Scalekit environment URL
   - `aud` / audience — if you rely on audience restriction for your API

6. **Enforce authorization** using application claims such as `scopes` (see [Scopes for machine clients](/authenticate/m2m/scopes/)).

</Steps>

<Aside type="tip" title="Prefer the Scalekit SDK when possible">
  Scalekit SDKs include helpers that validate access tokens against JWKS (issuer, signature, and optional scope checks). See the [M2M API authentication quickstart](/authenticate/m2m/api-auth-quickstart/) and [Authenticate customer apps](/guides/m2m/api-auth-m2m-clients/) for examples. You can still use generic JWT + JWKS libraries when you need custom middleware or a non-supported runtime.
</Aside>

## Operational practices

- **Cache JWKS responses** (many libraries do this by default). Keys rotate rarely; when they do, a new `kid` appears—refresh or let the client refetch on verification failure.
- **Fail closed**: invalid signature, wrong issuer, or expired token should return `401` (or `403` when claims are valid but insufficient).
- **Do not** accept tokens from unknown issuers or skip signature verification based on payload alone.
