Use this guide when you build a **companion CLI or SDK** that calls the same MCP server you already protect with Scalekit OAuth. MCP hosts such as Claude Desktop handle OAuth automatically; a CLI must implement the authorization code flow with **PKCE** itself.

## Prerequisites

- A Scalekit MCP server with OAuth enabled
- **Dynamic client registration (DCR)** enabled if you want the CLI to self-register (recommended for local dev)
- A loopback redirect URI (for example, `http://127.0.0.1:8765/callback`) registered or accepted by your authorization server

## Choose a client registration strategy

| Strategy | When to use |
|----------|-------------|
| **Dynamic client registration** | Local CLI tools, developer sandboxes, clients that cannot ship a static `client_secret` |
| **Pre-registered confidential client** | Production CLI distributed to a known set of operators; secret stored in a secure vault |

DCR is documented for MCP servers in [Additional reading](/authenticate/mcp/additional-reading/#dynamic-client-registration). Your CLI registers once, stores the issued `client_id`, and reuses it on subsequent runs.

<Aside type="caution" title="Never embed client secrets in CLI binaries">
  Public CLI distributions cannot keep a `client_secret` confidential. Use PKCE with a public client, or proxy token exchange through your backend.
</Aside>

<Steps>
1. ## Discover authorization server metadata

   Fetch your Scalekit OpenID configuration to locate `authorization_endpoint`, `token_endpoint`, and `registration_endpoint` (when DCR is enabled):

   ```bash
   curl -s "https://<YOUR_ENVIRONMENT_URL>/.well-known/openid-configuration" | jq '{authorization_endpoint, token_endpoint, registration_endpoint}'
   ```

2. ## Register the CLI (optional DCR)

   When using DCR, POST a client metadata document. Persist the returned `client_id` in a local config file outside version control.

   ```bash
   curl -s -X POST "https://<YOUR_ENVIRONMENT_URL>/oauth/register" \
     -H "Content-Type: application/json" \
     -d '{
       "client_name": "my-mcp-cli",
       "redirect_uris": ["http://127.0.0.1:8765/callback"],
       "grant_types": ["authorization_code", "refresh_token"],
       "response_types": ["code"],
       "token_endpoint_auth_method": "none"
     }'
   ```

3. ## Start authorization with PKCE

   Generate a `code_verifier` and `code_challenge` (S256). Open the authorization URL in the system browser or embedded web view.

   <Tabs syncKey="tech-stack">
   <TabItem label="Node.js">
   ```typescript
   import crypto from 'node:crypto'

   const verifier = crypto.randomBytes(32).toString('base64url')
   const challenge = crypto
     .createHash('sha256')
     .update(verifier)
     .digest('base64url')

   const params = new URLSearchParams({
     response_type: 'code',
     client_id: process.env.MCP_CLI_CLIENT_ID!,
     redirect_uri: 'http://127.0.0.1:8765/callback',
     scope: 'openid profile offline_access',
     code_challenge: challenge,
     code_challenge_method: 'S256',
     state: crypto.randomBytes(16).toString('hex'),
   })

   const authUrl = `${process.env.SCALEKIT_ENV_URL}/oauth/authorize?${params}`
   // Open authUrl in the user's browser; listen on 127.0.0.1:8765 for the callback
   ```
   </TabItem>
   <TabItem label="Python">
   ```python
   import base64
   import hashlib
   import os
   import secrets

   verifier = base64.urlsafe_b64encode(secrets.token_bytes(32)).rstrip(b"=").decode()
   challenge = base64.urlsafe_b64encode(
       hashlib.sha256(verifier.encode()).digest()
   ).rstrip(b"=").decode()

   # Build authorization URL with code_challenge and code_challenge_method=S256
   # Open in browser; capture ?code= on http://127.0.0.1:8765/callback
   ```
   </TabItem>
   </Tabs>

4. ## Exchange the code for tokens

   POST to the token endpoint with `grant_type=authorization_code`, the `code`, `redirect_uri`, `client_id`, and `code_verifier`. Request `offline_access` when you need refresh tokens.

   ```bash
   curl -s -X POST "https://<YOUR_ENVIRONMENT_URL>/oauth/token" \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "grant_type=authorization_code" \
     -d "code=<AUTH_CODE>" \
     -d "redirect_uri=http://127.0.0.1:8765/callback" \
     -d "client_id=<CLIENT_ID>" \
     -d "code_verifier=<CODE_VERIFIER>"
   ```

5. ## Call your MCP server

   Attach the access token as a Bearer token on MCP HTTP requests. Refresh before expiry using the refresh token when `offline_access` was granted.

</Steps>

## Troubleshooting

- **Popup or browser closes immediately** — If your CLI opens a browser popup through a web view, check for `Cross-Origin-Opener-Policy: same-origin` on your login page. See [Troubleshooting MCP auth](/authenticate/mcp/troubleshooting/#oauth-popup-closes-immediately-or-shows-window-closed-too-soon).
- **No refresh token** — Add `offline_access` to the scope list in the authorization request.
- **Invalid redirect URI** — Loopback URIs must match exactly between registration, the authorize request, and the token exchange.
