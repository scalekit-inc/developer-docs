Before activating a connected account, Scalekit confirms that the user who completed the OAuth consent is the same user your app originally intended to connect. This **user verification** step runs every time a connected account is authorized.

You can configure how verification works in **Agent Actions > User Verification** in the dashboard. Scalekit supports two modes:

![Agent Actions User Verification showing Custom user verifier and Scalekit users only](@/assets/docs/agent-actions/user-verification-config.png)

### Scalekit user verifier

Scalekit handles verification automatically, no code changes needed on your side.

For this to work, the user authorizing the connection must already be logged in to the Scalekit dashboard. If they are, verification completes using the logged in user. If they are not logged in, Scalekit displays an error page instead.

<Aside type="note" title="Dashboard access required">
  Users must be part of your Scalekit workspace and signed in to the dashboard before they can authorize connections. Switch to the Custom user verifier when you're ready to onboard users who don't have Scalekit accounts.
</Aside>

### Custom user verifier

Required for production. Your application takes over the verification step, so end-users never interact with Scalekit directly.

After the OAuth consent completes, Scalekit redirects the user's browser to a verify route you define, passing an `auth_request_id` and `state` as query parameters. Your route identifies the logged-in user, then calls Scalekit's verify API with the `auth_request_id` and the user's `identifier`. Scalekit matches the identifier against the one recorded when the flow started and, on success, activates the connected account.

## Build a Custom user verifier

<Steps>
1. ## Set up your environment

   Install the Scalekit SDK and initialize the client with your API credentials from **Dashboard > Developers > Settings > API Credentials**:

   <Tabs syncKey="tech-stack">
     <TabItem label="Python">
   ```sh showLineNumbers=false frame="none"
   pip install scalekit-sdk-python
   ```
   ```python showLineNumbers=false frame="none"
   import os
   from scalekit import ScalekitClient

   scalekit_client = ScalekitClient(
       env_url=os.getenv("SCALEKIT_ENV_URL"),
       client_id=os.getenv("SCALEKIT_CLIENT_ID"),
       client_secret=os.getenv("SCALEKIT_CLIENT_SECRET"),
   )
   ```
     </TabItem>
     <TabItem label="Node.js">
   ```sh showLineNumbers=false frame="none"
   npm install @scalekit-sdk/node
   ```
   ```typescript showLineNumbers=false frame="none"
   import { ScalekitClient } from '@scalekit-sdk/node';

   const scalekit = new ScalekitClient(
     process.env.SCALEKIT_ENV_URL!,
     process.env.SCALEKIT_CLIENT_ID!,
     process.env.SCALEKIT_CLIENT_SECRET!,
   );
   ```
     </TabItem>
   </Tabs>

2. ## Create the magic link with verification params

   Pass these fields when creating the magic link to enable verification:

   | Field | Description |
   |---|---|
   | `identifier` | **Required.** The user's identifier, e.g. email address or user ID. Scalekit stores this at link creation and checks it matches at verify time. |
   | `user_verify_url` | **Required.** Your application callback URL that Scalekit redirects the user to after OAuth completes. |
   | `state` | **Recommended.** A value you generate to correlate the callback to your session, the same way OAuth state works. Encrypt this before sending, as it travels through the browser as a query parameter. |

   <Aside type="note" title="How to use state">
     `state` works the same way as in standard OAuth/OIDC flows. Generate a cryptographically random value per flow, store it in a secure, HTTP-only cookie, and validate it when your verify endpoint receives the redirect. If the cookie value doesn't match the `state` query param, discard the request without calling the verify API. This prevents a malicious actor from sending a crafted verify URL to one of your users to hijack their session context.
   </Aside>

   <Tabs syncKey="tech-stack">
     <TabItem label="Python">
   ```python
   import secrets

   # Generate a state value to prevent CSRF
   state = secrets.token_urlsafe(32)
   # Store state in a secure, HTTP-only cookie to validate on callback

   response = scalekit_client.actions.get_magic_link_for_connected_account(
       connector=connector,          # e.g. 'gmail'
       identifier=user_id,           # current user id or email
       user_verify_url="https://app.yourapp.com/user/verify",
       state=state,
   )
   ```
     </TabItem>
     <TabItem label="Node.js">
   ```typescript
   import crypto from 'node:crypto';

   // Generate a state value to prevent CSRF
   const state = crypto.randomUUID();
   // Store state in a secure, HTTP-only cookie to validate on callback

   const { link } = await scalekit.actions.getMagicLinkForConnectedAccount({
     userId,                        // current user id or email
     connector,                    // e.g. 'gmail'
     userVerifyUrl: 'https://app.yourapp.com/user/verify',
     state,
   });
   ```
     </TabItem>
   </Tabs>

3. ## Build the verify route

   After OAuth completes, Scalekit redirects the user to your `user_verify_url`:

   ```
   GET https://app.yourapp.com/user/verify?auth_request_id=req_xyz&state=<your_state>
   ```

   Read the `state` from the cookie set earlier and compare it with the `state` query param. If they match, call Scalekit's verify endpoint with the `auth_request_id` and the user's `identifier`.

   <Aside type="caution" title="Never trust query params for identity">
     Read the user's identity from your own session, not from the URL. Use `state` for session correlation only.
   </Aside>

4. ## Confirm the user's identity

   Call the verify endpoint server-side using your client credentials:

   <Tabs syncKey="tech-stack">
     <TabItem label="Python">
   ```python
   # Get auth_request_id and state from URL query params
   # Read the stored state from the cookie

   # 1. Validate the state query param matches the state stored in the cookie
   # 2. Call Scalekit to verify the user identity
   response = scalekit_client.actions.verify_connected_account_user(
       auth_request_id=auth_request_id,
       identifier=user_id,    # must match what was stored at link creation
   )
   print(f"Redirect user to: {response.post_user_verify_redirect_url}")
   ```
     </TabItem>
     <TabItem label="Node.js">
   ```typescript
   // Get auth_request_id and state from URL query params
   // Read the stored state from the cookie

   // 1. Validate the state query param matches the state stored in the cookie
   // 2. Call Scalekit to verify the user identity
   const { postUserVerifyRedirectUrl } =
     await scalekit.actions.verifyConnectedAccountUser({
       authRequestId: auth_request_id,
       identifier: userId,     // must match what was stored at link creation
     });
   console.log('Redirect user to:', postUserVerifyRedirectUrl);
   ```
     </TabItem>
   </Tabs>

   On success, the connected account is activated. Redirect the user to your own success page or use the `post_user_verify_redirect_url` returned by Scalekit.

</Steps>

## How it works

```d2
shape: sequence_diagram

Your app -> Scalekit: POST magic_link (identifier, state, user_verify_url)
Scalekit -> Your app: magic link URL
Your app -> User: deliver link

User -> Scalekit: click magic link
Scalekit -> Google OAuth: redirect
Google OAuth -> Scalekit: auth code
Scalekit -> Scalekit: hold tokens pending verification
Scalekit -> User: redirect to user_verify_url

User -> Your app: GET /user/verify/callback?auth_request_id=...&state=...
Your app -> Your app: validate state and look up user id from session
Your app -> Scalekit: POST verify (auth_request_id, identifier)
Scalekit -> Scalekit: assert identifier matches, activate account
Scalekit -> Your app: respond with post_user_verify_redirect_url
Your app -> User: redirect to success page
```
